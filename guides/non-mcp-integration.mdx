---
title: Non-MCP integration
description: Add Sector8 enforcement to a containerized Codex CLI or other non-MCP tool caller with one evaluation endpoint.
---

Non-MCP tool callers -- including containerized Codex CLI deployments -- use
`POST /api/v1/evaluate` as their enforcement gate. Every tool call passes through
the same Admission Controller that governs the MCP Server. The container pattern
requires no code changes to the base image.

## Prerequisites

- A Sector8 API key and client ID (from the [Sector8 dashboard](https://app.sector8.ai))
- A container build that you control
- `pip` available in the container

---

## Step 1 -- Add Sector8 to the container image

Install the SDK once in your approved base image:

```dockerfile
FROM your-registry/codex-cli-base:latest

RUN pip install sector8-sdk

COPY sector8_policy.yaml /etc/sector8/policy.yaml

ENV SECTOR8_API_URL=https://sdkapi.sector8.ai
```

Do not bake credentials into the image. Inject them at runtime via the environment.

## Step 2 -- Set required environment variables

| Variable | Required | Purpose |
| --- | --- | --- |
| `SECTOR8_API_URL` | yes | Sector8 API base URL |
| `SECTOR8_API_KEY` | yes | tenant API key for `POST /api/v1/evaluate` |
| `SECTOR8_CLIENT_ID` | yes | tenant or client identifier |
| `CODEX_USER_ID` | yes | developer identity recorded in telemetry |
| `CODEX_ROLE` | yes | authorization role, for example `developer` |
| `CODEX_SESSION_ID` | yes | per-session trace and rate-limit identity |

## Step 3 -- Route tool calls through the adapter

Replace direct tool dispatch with the Sector8 adapter. The adapter calls
`POST /api/v1/evaluate` and raises `ToolBlockedError` on `DENY`:

```python
from sdk.adapters.openai_tools_adapter import OpenAIToolsAdapter, ToolBlockedError
import os

adapter = OpenAIToolsAdapter(
    api_url=os.environ["SECTOR8_API_URL"],
    api_key=os.environ["SECTOR8_API_KEY"],
    client_id=os.environ["SECTOR8_CLIENT_ID"],
    caller_id=os.environ["CODEX_USER_ID"],
    role=os.environ.get("CODEX_ROLE", "developer"),
    session_id=os.environ["CODEX_SESSION_ID"],
)
```

Route every tool call through `adapter.execute(...)` before local dispatch:

```python
try:
    result = adapter.execute(tool_name="bash", arguments={"command": cmd})
    # dispatch locally
except ToolBlockedError as e:
    print(f"Blocked: {e.reason_code}")
    print(f"Evidence hash: {e.evidence_hash}")
    print(f"Trace ID: {e.decision_trace_id}")
```

## Success state

When correctly wired:

- Allowed calls execute locally after an `ALLOW` decision.
- Denied calls raise `ToolBlockedError` and never reach the OS.
- Every call -- ALLOW and DENY -- appears in your Sector8 telemetry.
- DENY decisions trigger the alert dispatcher if a webhook or SNS ARN is configured.

## Evaluation response schema

```json
{
  "decision": "ALLOW|DENY",
  "reason_code": null,
  "evidence_hash": "",
  "has_forensic_payload": false,
  "decision_trace_id": "uuid",
  "policy_version_id": "string"
}
```

On `DENY`, `reason_code` is populated and `has_forensic_payload` is `true`.

## Gotchas

- **`session_id` is required for rate limiting.** A missing or static `session_id` will cause all requests to share a rate limit bucket.
- **`owned_objects` must list what the caller actually owns.** BOLA checks compare `object_id` against this list -- an empty list means every object-level request is denied.
- **Do not cache `ALLOW` decisions.** The Admission Controller evaluates every call independently. A cached `ALLOW` is not a security property.

## Honest scope

This path governs tool calls that route through the adapter. It does not govern inline
completions or chat content that never becomes a governed tool call.
